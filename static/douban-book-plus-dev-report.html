01 · Executive Summary

执行摘要

本文档记录了一款 豆瓣读书增强 Chrome 插件 的完整开发过程——从项目初始化到七个在线阅读平台的直达链接注入。核心工作是 五个平台的搜索 API 逆向工程，涉及 JavaScript MD5 编码算法移植、SPA 页面 JS Bundle 源码挖掘、CORS 跨域策略绕过等多个技术难点。

核心 Takeaways

  • 七个平台全部接入插件面板，其中五个平台实现了书名→直达详情页的智能解析
  • API 逆向方法论从被动等待用户提供→主动探测每个平台的搜索端点
  • 微信读书的 MD5 编码算法需要从开源社区（obsidian-weread-plugin）移植纯 JS 实现
  • 网易蜗牛读书的搜索接口藏在官网 web_intro/index.js 的 SPA bundle 中，非标准 REST 路径
  • 图标系统从 emoji 迁移至 SVG 内联 data URI，自包含无网络依赖
本文档回答的问题

如何在不解包 APK 的情况下，通过抓取 Web 端资源完成七个不同阅读平台的搜索 API 逆向？
一个 Manifest V3 Chrome 插件的 Service Worker → Content Script 消息通道如何设计才能稳健处理异步解析？
哪些坑（CORS、精度丢失、重定向吞噬参数）会在实际部署时反咬一口？

02 · Background

项目背景与目标

灵感来源于 Edge 浏览器的 Douban Book+ 插件：在豆瓣读书页面注入一个侧边面板，展示该书在多个主流阅读平台上的搜索链接。我们的目标是实现一个功能更强、用户体验更好的 Chrome 版本——从"搜索页兜底"升级为"直达详情页"。

为什么做这个插件

bitbetter 运营公众号 iBitBetter，日常阅读量大，经常需要在豆瓣读书发现好书后，快速跳转到不同平台阅读。每换一个平台就重新搜一遍书名，流程繁琐且低效。

市面上已有 Edge 版 Douban Book+，但它只提供各平台的搜索页链接——用户仍需在目标平台手动点击搜索结果。我们决定做一层升级：直接跳到书的详情页或阅读页，一步到位。

豆瓣阅读 豆 直达 reader/ebook/{id}/

五个直达平台中，微信读书的编码算法是最复杂的技术难点——需要从开源插件中移植纯 JavaScript 的 MD5 哈希实现。网易蜗牛的搜索 API 是隐藏最深的——藏在官网的 web_intro/index.js 这个不明显的 SPA Bundle 文件中。Z-Library 则因为镜像站的跳转机制，无法携带查询参数，只能退化为首页链接。

03 · Architecture

技术架构

插件采用 Manifest V3 标准架构，核心通信模型为：豆瓣页面 → Content Script → Service Worker（background.js）→ 目标平台 API → 返回直达链接。异步解析通过 chrome.runtime.sendMessage 实现。

项目结构

douban-book-plus/
├── manifest.json            # Manifest V3 配置，host_permissions
├── content.js               # 内容脚本：DOM 注入 + 面板UI构建
├── content.css              # 面板样式（与豆瓣原生风格融合）
├── background.js            # Service Worker：API 调用 + 链接解析
└── utils/
    └── weread-encode.js      # 纯 JS MD5 实现 + 微信读书加密算法

通信流程

当用户访问豆瓣读书页面（book.douban.com/subject/*）时：

  1. Content Script 从页面 DOM 中提取书名和 ISBN
  2. 对标记了 needsResolve: true 的平台，通过 chrome.runtime.sendMessage 向 Service Worker 发起异步请求
  3. Service Worker 收到 RESOLVE_WEREAD_URL / RESOLVE_DUOKAN_URL 等消息后，调用对应平台的搜索 API
  4. 解析成功后返回 {url, found: true}，Content Script 更新链接并添加"直达"徽标
  5. 失败时保持搜索页链接作为兜底

为什么用 Service Worker 而非 Content Script 直接 fetch？

Manifest V3 的 Service Worker 拥有扩展级特权，不受目标网站的 CORS 策略限制。实测中，网易蜗牛搜索 API 从 Content Script 直接 fetch 会被 CORS 拦截，而通过 Service Worker 中继则正常返回。

异步解析状态机

面板于 DOM 构建阶段 一次性渲染全部平台条目，每个条目预留 asyncItems[platformId] 引用。各解析器异步返回结果后，仅更新对应条目的 href 和 title，并注入"直达"徽标。不会出现"面板先空白再慢慢出现"的体验问题。

04 · Reverse Engineering

五平台 API 逆向工程

五个直达平台的搜索 API，没有一个提供了公开文档。以下按逆向难度从高到低逐一记录发现过程。

4.1 微信读书 — MD5 编码的黑箱

直接链接格式：https://weread.qq.com/web/bookDetail/ce032b305a9bc1ce0b0dd2a。其中 ce032b305a9bc1ce0b0dd2a 并非常规 bookId，而是经过编码后的结果。

通过搜索 obsidian-weread-plugin 开源项目，定位到其 src/utils.ts 中暴露的编码逻辑：

  1. 搜 web/search/global?keyword=书名 → 返回原始 bookId（如 "3300132100"）
  2. 对 bookId 做 标准 MD5 哈希
  3. 将 MD5 hash（32 位十六进制）按相邻两位一组转为十进制数（共 16 组），每组对 62 取模，映射到 0-9a-zA-Z 的 codeMap 表
  4. 取转换结果的前三位作为校验码，拼接到原 bookId 后，得到最终 encodedId

在 utils/weread-encode.js 中完整实现了该算法——包括一个 纯 JavaScript 的 MD5 哈希实现（因 Service Worker 环境无法使用 Node.js crypto 模块）。调试过程中发现标准 MD5 输出与预期不符，通过交叉验证修复了字节序问题。

// 伪代码摘要
function encodeWereadId(bookId) {
  var md5Hex = md5(bookId);           // 标准 MD5 → 32 位 hex
  var str = hexToCharCodes(md5Hex);   // 每两位 hex → 十进制码点
  var encoded = mapToCode62(str);     // 对 62 取模映射到 [0-9a-zA-Z]
  return bookId + encoded.slice(0, 3); // bookId + 前三位校验
}

4.2 豆瓣阅读 — 最普通的 JSON API

直接链接格式：https://read.douban.com/reader/ebook/15077436/

端点参数关键字段认证
GET /j/search?query=书名data[].id（需过滤 type==="ebook"）无需

豆瓣阅读内部的 /j/search 接口返回所有类型的结果（整书、单篇、专栏等），需过滤 type === "ebook" 才取到正确的整书 ID。匹配逻辑优先取第一个类型匹配的结果。

这是五个平台中仅次于多看最简单的对接——一个标准 RESTful JSON API，无身份认证，无编码变换。

4.3 得到 — 绕过登录门槛

直接链接格式：https://www.dedao.cn/ebook/reader?id=YxRj1dbLGNgA2oaD6VBevmQZ7rnbYWmz7n3kyOKRMzJpX9lP5dxqE41j8De2pmOP

端点方法参数关键字段认证
/api/search/pc/suggestPOST{"query":"书名","searchType":2}data[].extra.enid（需过滤 type===2）无需

得到的搜索 API 存在两个显著特点：

  • POST 而非 GET：需用 JSON body 传参，searchType=2 限定只搜电子书
  • enid 字段非标准：book ID 藏在 extra.enid 而非顶层字段，极易忽略

一开始尝试了 /ent-search/ 接口，返回 401 要求登录。转而探测 suggest API —— 这是少数几个对匿名用户开放的端点之一。仅支持按书名搜索，无法按 ISBN 匹配。

4.4 多看阅读 — 参数名称的秘密

直接链接格式：https://www.duokan.com/reader/www/app.html?id=91e8d81e816c4612aebed9b9505a3682

端点参数关键字段认证
GET /target/search/web?s=书名&p=1books[0].book_id无需

搜索参数用的是 s 而非常见的 q / keyword / wd。最初所有常规参数名均返回空结果，直到翻阅多看移动端的 JS 源码，才在 API 调用行找到了 s 这个参数名。这个发现说明了为什么凭空猜测 API 参数是不可靠的——API 逆向必须依赖静态代码分析而非枚举试错。

4.5 网易蜗牛读书 — 藏在 SPA Bundle 里的 API

直接链接格式：https://du.163.com/share/book/7525592637001275380

网易蜗牛是五个平台中 API 隐藏最深 的一个，其 du.163.com 是纯 SPA，所有服务端路由均返回"内容不存在"，必须通过客户端 JS 渲染。搜索 API 被埋在官网的 web_intro/index.js 这个不显眼的 Bundle 文件中。

端点参数关键字段认证
GET /search/book.json?word=书名&page=1&pageSize=3bookWrappers[0].book.bookId无需

这个接口有两个隐藏陷阱：

  • 不支持 ISBN 搜索：用 ISBN 查询返回空数组，只能用书名
  • bookId 精度丢失：返回的 bookId 是 19 位数字（如 7525592637001275380），超过 JS Number.MAX_SAFE_INTEGER（约 9×10¹⁵），直接 JSON.parse 会截断末尾。解决方案是优先取 book.bookId（字符串字段）而非顶层 bookWrappers[0].bookId（数字字段）

精度丢失演示：JSON.parse('[7525592637001275380]') 在控制台输出 [7525592637001275000]——末尾三位完全错误，导致生成的分享链接失效。

05 · Pitfalls

关键踩坑记录

以下五个问题是开发过程中最耗时的故障点。每个都经历了"表现正常 → 部署失效 → 日志排查 → 根因定位"的完整循环。

坑 1：CORS 跨域拦截（网易蜗牛）

表现：网易蜗牛搜索在 Content Script 中直接 fetch 时报错 blocked by CORS policy。

根因：du.163.com 的 OPTIONS 预检对来自 book.douban.com 的请求直接返回 403。

修复：回归 Service Worker 中继方案。Manifest V3 的 Service Worker 拥有扩展级网络权限，不受目标站 CORS 策略限制。

教训：Chrome 插件的 Content Script 和 Service Worker 有不同的网络权限边界。Content Script 受限于宿主页面的同源策略，而 Service Worker 处于扩展的独立环境中。API 调用始终应通过 Service Worker 中继。

坑 2：ISBN 搜索失效（网易蜗牛）

表现：网易蜗牛始终显示无搜索结果，返回 found: false。

根因：代码中写了 isbn || title，ISBN 有值时优先使用。但网易蜗牛搜索 API 不支持 ISBN 查询，用 ISBN 返回空 bookWrappers: []。

修复：强制使用书名搜索：var searchQuery = title;。

坑 3：JavaScript Number 精度溢出（网易蜗牛）

表现：生成的分享链接 /share/book/7525592637001275000 404 打不开。

根因：网易蜗牛 bookId 是 19 位整数，超过 JS 安全整数范围。API 返回的 JSON 中该字段以数字形式出现在顶层对象，JSON.parse 后丢失末尾精度。

修复：JSON 响应中 book.bookId 是字符串类型，不会被 JSON.parse 转换为 Number。代码改为优先从 bookWrappers[0].book.bookId 取值。

教训：处理大于 16 位的数字 ID 时，永远优先取字符串字段。可在 API 探测阶段就用 typeof 检查关键 ID 的类型。

坑 4：302 重定向吞噬参数（Z-Library）

表现：http://zlib.re/s/9787545493474 跳转后丢失搜索词。

根因：zlib.re 是镜像站，先 302 跳转到真实域名（如 zh.vbh101.ru），跳转过程中 路径和查询参数全部被服务器丢弃。

修复：链接指向镜像首页 http://zlib.re/，不做参数传递。手动搜索反而比自动跳转更可靠。

坑 5：Service Worker 缓存不更新

表现：修改 background.js 后，Chrome 仍运行旧版代码。

根因：Chrome 的 Service Worker 生命周期独立于页面——"刷新扩展"（Reload）不总能触发 Worker 重新加载。

修复：先移除扩展再重新加载（"加载已解压的扩展程序"重新选文件夹）。或者在 chrome://serviceworker-internals/ 中手动 Unregister 残留 Worker。

图标系统升级

初版使用 emoji 作为平台图标（📖📗📘📙🐌📚🏛️），但在深色模式和不支持 emoji 的平台上显示效果不稳定。改用 SVG 内联 data URI 方案：每个平台用对应品牌色生成圆角方块 + 首字（微/豆/得/多/蜗/Z/A），通过 makeIcon(color, letter) 工具函数动态生成。完全自包含，零网络依赖。

function makeIcon(color, letter) {
  var svg = '<svg xmlns="http://www.w3.org/2000/svg"
    viewBox="0 0 24 24">' +
    '<rect width="24" height="24" rx="5" fill="' + color + '"/>' +
    '<text x="12" y="17" text-anchor="middle"
    font-size="14" fill="white">' + letter + '</text>' +
    '</svg>';
  return 'data:image/svg+xml,' + encodeURIComponent(svg);
}
06 · Conclusions

结论与后续规划

七个平台全部接入，五个实现直达链接解析。核心技术收获集中在 API 逆向方法论和 Chrome 插件异步通信模式的工程化实践。

核心结论

  1. API 逆向的关键路径：SPA JS Bundle → 原域名探测 → GitHub 开源项目参考。凭空猜测参数名成功率极低，必须依赖静态代码分析
  2. Manifest V3 的 Service Worker 是中继 API 调用的最佳位置，同时提供扩展级网络权限和独立于宿主页面的执行环境
  3. 跨平台搜索 API 的共性差异需在每个平台单独处理：参数名（q/keyword/s/word）、搜索范围（ISBN 支持）、结果字段结构（提取路径），无法模板化
  4. 精度丢失是无声杀手：JS Number 安全范围限制需要在使用大整数 ID 时强制取字符串字段

后续规划

  • Z-Library 直达：探索通过 WebSocket 或 POST 方式绕过 302 重定向限制，或在客户端维护镜像列表实现智能跳转
  • 缓存层：Service Worker 端增加内存缓存（Map 存储书名→bookId 映射），避免重复请求同一本书
  • 国际化：支持多语言书名搜索（如英文书名匹配中文平台）
  • 发布 Chrome Web Store：合规审查后提交审核
输出物

Chrome 插件源码：douban-book-plus/（Manifest V3 完整项目）
可直接加载为"未打包的扩展程序"使用，无需额外依赖。
